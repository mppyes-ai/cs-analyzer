## 问题描述

### 子问题 1: 文件名显示错误
用户分析 `客服聊天记录(20).log`，但进度更新显示的是之前残留的 `客服聊天记录(10).log` 的进度信息，随后才突然切换回正确的文件名。

### 子问题 2: 误导性"分析已完成"提示
在后台模式下，系统在分析尚未完成时就输出了"分析已完成"、"正在查询详细结果"等误导性信息，导致用户以为分析已经结束，实际上 Worker 仍在后台运行。

## 修复历史

### 第一轮修复 (2026-04-03)

**问题 1 修复方案**: 启动时清理旧消息文件

**修改文件**: `batch_analyzer.py`

**关键代码**:
```python
def _clear_message_files(self):
    """清理旧消息文件，避免残留消息干扰新分析"""
    import os
    msg_file = '/tmp/cs_analyzer_messages.jsonl'
    processed_file = '/tmp/cs_analyzer_messages_processed.jsonl'
    
    for f in [msg_file, processed_file]:
        if os.path.exists(f):
            os.unlink(f)
            print(f"  🧹 清理旧消息文件: {f}")
```

**调用位置**:
- `run_background()` 和 `run_foreground()` 开头调用

## 根因分析

### 问题 1 根因
**残留消息机制**:
```
分析 A.log:
  ├─ monitor_agent 写入: "进度: 10%..."
  ├─ monitor_agent 写入: "进度: 20%..."
  └─ 分析完成，但消息文件未清空

分析 B.log (新任务):
  ├─ message_poller 仍在运行（或重启后）
  ├─ 读取残留消息: "进度: 10%..."（显示A的文件名）
  ├─ 用户困惑：我在分析B，为什么显示A？
  └─ 直到新消息覆盖才显示正确
```

### 问题 2 根因
**交互流程设计缺陷**:
```
旧流程:
1. 用户发起分析
2. 系统启动后台Worker
3. [错误] 立即查询数据库输出"分析已完成"
4. [错误] 继续查询并输出中间报告
5. 系统推送真实进度（10%、20%...）
6. 用户困惑：到底完成了没有？

新流程:
1. 用户发起分析
2. 系统返回启动状态（无误导信息）
3. 系统推送真实进度（10%、20%...）
4. 系统推送完成报告（含详细统计）
5. 用户体验：清晰、无歧义
```

## 经验总结

**设计原则**:
1. **状态清晰原则**: 后台模式启动后只应返回"已启动"状态，不应返回中间结果
2. **单一职责原则**: 进度推送由 monitor_agent 负责，不应由主进程重复查询
3. **清理原则**: 临时文件在会话开始前应清理，避免残留干扰

**可改进点**:
- 可使用唯一的 batch_id 隔离不同分析的消息文件（当前使用全局文件）
- 可考虑使用 Redis / RabbitMQ 替代文件消息队列（生产环境）
