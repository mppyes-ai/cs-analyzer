## 问题描述

### 现象
用户明确发起**前台模式**分析时，出现以下异常：
1. 进度直接跳到 100%
2. 显示 "已完成: 23/5 通"（明显逻辑错误）
3. 实际分析并未完成，但进度显示已完成

### 用户反馈
> "第二轮会话明确发起要求前台模式分析,也正常进入前台分析，然而'📈 CS-Analyzer 进度更新'有明显问题,直接就显示100%完成,同时'已完成:23/5通'也有明显问题,并且实际分析并未完成"

### 预期行为
- 前台模式应**不推送**飞书进度消息（前台阻塞运行，自己在控制台显示进度）
- 进度应从 0% → 10% → ... → 100% 逐步显示
- "已完成: x/total 通" 应准确反映实际进度

## 修复历史

### 第一轮修复 (2026-04-03)

**问题1根因**: 
前台模式错误地启动了 `monitor_agent`，导致：
1. `monitor_agent` 后台推送飞书进度消息
2. 前台控制台也显示进度
3. 双重进度显示，数据不一致

**问题2根因**:
前台模式的进度计算逻辑不一致：
- `completed_this_batch`: 只统计当前批次完成的任务（通过遍历 task_ids 查询）
- `pending/processing`: 从 `get_queue_stats()` 获取的是**全局**状态

当数据库中有其他历史任务时，`completed_this_batch` 与全局状态不一致，导致计算错误。

**问题代码**:
```python
# batch_analyzer.py:run_foreground()

# 4. 启动监控代理（前台模式也推送进度）  <-- 错误！前台不应启动
if result['submitted'] > 0:
    self.spawn_monitor_agent(...)

# ... 轮询等待 ...
# 只统计本次任务的完成情况
completed_this_batch = sum(1 for tid in result['task_ids'] 
                           if get_task_detail(tid)['status'] == 'completed')
# 但 pending/processing 是全局的！
if pending == 0 and processing == 0 and completed_this_batch >= total:
```

**修复方案1**: 前台模式不启动监控代理

**修改文件**: `batch_analyzer.py`

**关键代码变更**:
```diff
-        # 4. 启动监控代理（前台模式也推送进度）
-        if result['submitted'] > 0:
-            self.spawn_monitor_agent(
-                total_tasks=result['submitted'],
-                log_file=log_file,
-                batch_id=''
-            )
+        # 4. 前台模式不启动监控代理（前台自己管理进度显示）
+        # 后台模式才需要监控代理推送进度
+        pass
```

**修复方案2**: 统一使用全局队列统计

**关键代码变更**:
```diff
-            # 只统计本次任务的完成情况
-            completed_this_batch = sum(1 for tid in result['task_ids'] 
-                                       if get_task_detail(tid) and 
-                                       get_task_detail(tid)['status'] == 'completed')
-            
-            # 计算进度
-            if total > 0:
-                progress = int((completed_this_batch / total) * 100)
-            else:
-                progress = 100
-            
-            # 打印进度
-            if progress != last_progress and progress % 10 == 0:
-                print(f"  进度: {progress}% ({completed_this_batch}/{total})")
+            # 计算进度（使用全局completed，前台模式独占运行）
+            if total > 0:
+                progress = int((completed / total) * 100)
+            else:
+                progress = 100
+            
+            # 打印进度（每10%显示一次）
+            if progress != last_progress and progress % 10 == 0:
+                print(f"  进度: {progress}% ({completed}/{total})")
```

**结果**: ✅ 修复成功，前台模式进度显示正确

**验证步骤**:
1. 确保队列中有历史任务（模拟多轮分析场景）
2. 发起前台模式分析 5 通会话
3. 确认：
   - 不收到飞书进度推送 ✅
   - 控制台进度从 0% → 10% → ... → 100% 逐步显示 ✅

## 根因分析

### 架构设计混淆
```
混淆点：前台/后台模式的职责边界不清晰

前台模式 (run_foreground):
  ├─ 阻塞等待
  ├─ 控制台输出进度
  ├─ 适合小批量、调试使用
  └─ 不应有飞书推送（用户就在终端前）

后台模式 (run_background):
  ├─ 立即返回
  ├─ monitor_agent 推送飞书进度
  ├─ 适合大批量、长时间运行
  └─ 需要飞书推送（用户可以离开）
```

### 历史代码问题
`completed_this_batch` 的计算逻辑是为了解决 H-CORE-006 的全局进度问题：
- H-CORE-006: 二次分析时全局 `completed` 包含历史任务，导致进度溢出 100%
- 修复方案: 记录 `initial_completed` 基准值，计算相对进度

但前台模式的修复引入了新的不一致：
- `completed_this_batch`: 通过 task_ids 遍历查询（准确但慢）
- `pending/processing`: 全局状态（可能包含其他批次任务）

前台模式是**独占式**运行的（用户等待，不会并发启动多个前台分析），因此可以直接使用全局统计。

## 经验总结

**设计原则**:
1. **模式职责清晰**: 前台/后台模式应有明确的职责边界
2. **避免双重输出**: 同一信息不应通过多个渠道同时推送
3. **状态一致性**: 进度计算应基于同一数据源，避免混合统计

**前台模式定位**:
- 用于小批量调试（< 100 通会话）
- 用户终端阻塞等待
- 控制台实时显示进度
- 不推送飞书消息

**后台模式定位**:
- 用于大批量生产（1000+ 通会话）
- 立即返回，后台运行
- 飞书推送进度和完成报告
- 支持用户关闭终端后继续运行
