# H-PERF-001 Worker批次内串行执行问题

**修复日期**: 2026-04-04  
**影响版本**: v2.5 及之前  
**修复版本**: v2.5.1  
**优先级**: High  
**架构层**: Performance

---

## 问题描述

Worker的 `_batch_score_with_limit_v2` 函数中，虽然配置了 `KIMI_MAX_CONCURRENT=50` 高并发，但实际执行时只有 **1个批次** 在处理，其他批次在等待。

### 根因分析

**错误代码模式**:
```python
for batch in batches:              # 串行循环
    async with kimi_semaphore:     # 锁在循环内！
        result = await score(...)  # 一次只能进一批
```

**问题**: `async with kimi_semaphore` 在 `for` 循环内部，导致：
- 批次1进入 → 获取锁 → 执行 → 释放锁
- 批次2进入 → 获取锁 → 执行 → 释放锁
- ...

实际并发度 = **1**，配置50完全无效。

---

## 修复方案

### 修改文件
- `worker.py` - `_batch_score_with_limit_v2()` 函数

### 修复后代码
```python
async def score_one_batch(batch_idx: int, batch: List[Dict]) -> List[Dict]:
    """评分单个批次（内部使用信号量限流）"""
    async with kimi_semaphore:     # 各批次独立竞争锁
        return await score(...)

# 全部同时启动，真正并行竞争50个并发槽位
await asyncio.gather(*[
    score_one_batch(i, batch) for i, batch in enumerate(batches)
])
```

**关键改动**: `for` 循环 + 内部锁 → `asyncio.gather` + 各任务独立竞争锁

## 经验教训

1. **信号量位置决定一切**: `async with` 在循环内 vs 循环外，效果天差地别
2. **asyncio.gather 是真正并行**: `for + await` 本质是串行
3. **配置不等于实际**: 配置50并发不代表真的会有50并发，需要代码正确实现
