## 🎯 核心结论对齐

### 三方共识 ✅

1. **问题性质**: 不是"4个固定坏数据"，而是"批次级不稳定"
2. **主因**: 本地模型(qwen3.6-35b-a3b)批量JSON输出不稳定
3. **放大器**: 重试链路没有真正降级，批量失败→批量重试→再次失败
4. **温度配置**: 本地模型硬编码temperature=1，导致输出漂移
5. **规则检索**: 长期返回0条规则，模型在无grounding情况下裸输出

### 分歧点 ⚠️

| 观点 | Opus | Codex | 本分析 |
|------|------|-------|--------|
| `.format()`花括号风险 | 认为是session_0023主因 | 认为是潜在炸点，非本轮主因 | **认同Codex**：本轮未被触发 |
| 4个会话失败频率 | 描述为"每次都失败(5/5)" | 纠正为"3/5失败" | **认同Codex**：实际3/5 |
| 失败类型 | 强调"结构漂移"为主因 | 指出"结构漂移+JSON坏格式"并存 | **认同Codex**：两类都存在 |

## 🔍 完整因果链（最终版）

```
规则检索100%返回0条
        ↓
模型在无规则grounding下处理异构会话（极短3条 + 正常 + 41条长会话混批）
        ↓
本地模型temperature=1（硬编码）→ 输出随机性高
        ↓
模型概率性返回：
  a) 单个dict而非list（结构漂移）
  b) 坏JSON格式（JSON decode error）
        ↓
_parse_batch_response()旧逻辑：用[result] + [{}...]补空占位
        ↓
batch_scoring.py检测到N-1条缺少dimension_scores → fail_task()
        ↓
worker.py重试：调用_retry_tasks_batch(batch_size=5) → 继续批量
        ↓
重试依然批量 → 依然结构漂移 → 依然N-1条失败 → 再次fail_task()
        ↓
retry_count达到3 → 永久失败
        ↓
哪些会话进入"倒霉批次"是随机的 → 每轮失败集不同 → 5轮结果波动
```

**第2轮100%成功解释**：批次切分恰好把问题会话分散，且其他会话内容足够强，模型在0规则情况下也保持了数组格式输出。**这是运气，不是能力**。

## 🛠️ 最终修复方案（已实施）

### P0-1: 重试链路强制降级为单通 ✅

**问题**: 结构性失败（评分结果不完整、JSON解析失败）重复走批量路径

**修复**:
- `task_fetcher.py`: 查询失败任务时带回`error`和`scene`字段
- `worker.py`: 新增`_smart_retry_failed_tasks()`，按错误类型分流
  - 结构性错误 → 单通并发重试
  - 非结构性错误 → 批量3通重试

**效果**: 切断"批量失败→批量重试→再次失败"的放大器

### P0-2: 本地模型温度读配置 ✅

**问题**: `smart_scoring_v2.py`硬编码`temperature=1`

**修复**:
- 同步/异步两条评分链路都改为`LLM_CONFIG.get("temperature", 0.1)`
- 本地模型使用0.1，云端Kimi使用1

**效果**: 降低本地模型输出随机性，提升JSON格式稳定性

### P0-3: 批量解析不再补空占位符 ✅

**问题**: `_parse_batch_response()`用`[result] + [{}...]`污染整批

**修复**:
- 返回带`_needs_single_retry`标记的降级信号
- `batch_scoring.py`识别标记，批内直接降级为单通处理
- 不再生成虚假的空占位符

**效果**: 一条坏结果不再污染整批

### P1-1: 会话风险分层 ✅

**问题**: 极短会话(≤5条)、长会话(≥41条)混入普通批次

**修复**:
- `batch_scoring.py`: 新增`SHORT_MSG_THRESHOLD=6`、`OVERSIZED_MSG_THRESHOLD=40`
- ≤5条消息 → 单通处理
- ≥41条消息 → 单通处理
- 普通会话 → 批量处理

**效果**: 消除异构会话相互干扰

### P1-2: 规则检索兜底 ✅

**问题**: 知识库rules表为空，模型裸输出

**修复**:
- `smart_scoring_v2.py`: 新增`GENERIC_RULES_FALLBACK`通用评分标准
- 规则为空时注入明确基准，而非静默继续
- 场景名统一标准化(`SCENE_ALIASES`)
- URL文本归一化为`[URL]`，降低链接干扰

**效果**: 模型有grounding，输出稳定性提升

### P2: `.format()`安全替换（待实施）

**问题**: `BATCH_SCORING_PROMPT_TEMPLATE.format(sessions_content=sessions_json)`存在潜在风险

**修复方案**:
```python
# 把sessions_json的花括号转义后再传入.format()
sessions_json_safe = sessions_json.replace("{", "{{").replace("}", "}}")
prompt = BATCH_SCORING_PROMPT_TEMPLATE.format(
    count=len(sessions),
    retrieved_rules=rules_text,
    sessions_content=sessions_json_safe
)
```

**优先级**: P2（本轮未被触发，但需预防）

## 💡 关键教训

1. **不要信任批量输出的稳定性**: 本地模型批量JSON输出不可靠，必须有降级机制
2. **重试必须真正降级**: 批量失败→批量重试是死亡循环，必须改为单通
3. **温度配置必须统一**: 硬编码temperature=1对本地模型是灾难
4. **规则grounding是基础**: 0条规则时模型裸输出，稳定性大幅下降
5. **会话分层是必要的**: 不要把3条消息和41条消息混在同一批
