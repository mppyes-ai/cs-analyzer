# oMLX 配置优化测试报告

## 一、测试背景

**日期**：2026-04-26  
**测试目标**：在批量处理大数据时保证输出质量的同时提升速度  
**硬件环境**：MacBook Pro M5 Max 128GB（614 GB/s内存带宽）  
**模型**：Qwen3.6-35B-A3B-4bit（20GB权重，MoE架构，每次激活约3B参数）  
**测试数据**：客服聊天记录400通（实际有效359通，过滤41通）  
**oMLX版本**：v1.0

## 四、关键发现

### 4.1 缓存调整是性价比最高的优化

**第1轮 → 第2轮**：仅调整缓存配置，速度提升22%
- 热缓存：关闭 → 50%
- 冷缓存：10% → 20%
- **效果**：24.6分钟 → 19.2分钟
- **评分质量**：保持稳定

**根因分析**：
- oMLX日志显示，SSD cache hit时速度可达85.4 tok/s（无缓存时26.4 tok/s）
- 热缓存保留最近50%的KV Cache，重复prompt前缀直接复用
- 冷缓存将溢出的KV Cache持久化到SSD，重启后恢复

### 4.2 TurboQuant无额外收益

**第2轮 → 第3轮**：开启TurboQuant KV Cache + 4bit
- 速度：持平（19.2分钟）
- 评分质量：微降（转化能力2.9→2.8）
- **结论**：在当前场景下TurboQuant收益有限

### 4.3 高并发在Apple Silicon上有害

**第2轮 → 第4轮**：oMLX并发32→128，Worker 30→120
- 速度：下降41%（19.2→27.1分钟）
- **根因**：M5 Max内存带宽614 GB/s，30+请求并发时每个请求仅5-8 tok/s
- 单请求无竞争时可达20-28 tok/s

### 4.4 Opus建议的保守配置过慢

**第2轮 → 第5轮**：oMLX 32→6，Worker 30→12，KIMI 90→4
- 速度：下降54%（19.2→29.6分钟）
- **根因**：KIMI=4严重限制并发，oMLX 6个灶台仅4个在用

### 4.5 中等并发未达最优

**第6轮**：oMLX=8，Worker=50，KIMI=30
- 速度：比最优慢26%（19.2→24.3分钟）
- **根因**：Worker=50产生50个HTTP请求，但KIMI=30限制为30个，oMLX=8仅处理8个，大量请求排队

## 五、最优配置

### 5.1 当前最优（截至2026-04-26）

| 组件 | 配置项 | 最优值 |
|------|--------|--------|
| **oMLX** | max-concurrent-requests | **32** |
| **oMLX** | hot_cache_max_size | **50%** |
| **oMLX** | ssd_cache_max_size | **20%** |
| **CS-Analyzer** | WORKER_MAX_GROUPS | **10** |
| **CS-Analyzer** | WORKER_MAX_WORKERS | **3** |
| **CS-Analyzer** | KIMI_MAX_CONCURRENT | **90** |
| **CS-Analyzer** | BATCH_SCORE_SIZE | **5** |

### 5.2 配置关系说明

```
Worker总并发 = 10组 × 3工作者 = 30并发
         ↓
KIMI=90（不限制，30 < 90）
         ↓
oMLX=32（处理32个请求，但Worker只有30个）
         ↓
实际并发 = min(30, 90, 32) = 30
```

**关键洞察**：
- KIMI=90是冗余保护，实际并发由Worker=30决定
- oMLX=32略大于Worker=30，确保GPU始终有活干
- 这是"超额订阅"（Over-subscription）策略：保持GPU满载

## 七、待验证优化方向

### 7.1 系统Prompt优化（高优先级）
- 将固定评分标准移入system prompt
- 利用oMLX前缀缓存机制
- **预估收益**：prefill时间减少60%，总耗时14-16分钟

### 7.2 代码级优化（中优先级）
- 将asyncio.as_completed改为严格串行
- 避免同时创建所有批次协程
- **预估收益**：减少请求堆积，提升稳定性

### 7.3 热缓存比例提升（低优先级）
- 尝试热缓存70-80%（如果内存允许）
- **预估收益**：进一步提升缓存命中率

---

## 八、测试结论

1. **缓存调整是性价比最高的优化**：简单调整缓存比例即可提升22%速度
2. **高并发在Apple Silicon上有害**：超过最优阈值后速度反而下降
3. **理论建议需实测验证**：Opus建议的保守配置实测慢54%
4. **当前最优配置**：oMLX 32并发 + 热缓存50% + 冷缓存20% + Worker 30并发
5. **下一步**：实施系统Prompt优化，预估再提升20-30%
