## 二、oMLX 缓存机制详解

### 2.1 什么是 KV Cache？

**核心概念**：LLM生成token时，需要"记住"之前生成的所有内容。KV Cache就是这份"记忆"。

```
生成过程：
第1个token → 计算量大（要处理整个prompt）
第2个token → 只需计算新token，复用第1个的缓存
第3个token → 复用第1+2的缓存
...
```

**没有KV Cache**：每生成1个token都要重新计算所有历史 → 速度慢100倍  
**有KV Cache**：只需计算新token → 速度提升100倍

### 2.2 热缓存（Hot Cache / 内存缓存）

| 属性 | 说明 |
|------|------|
| **存储位置** | 内存（RAM） |
| **速度** | 最快（与GPU共享统一内存） |
| **容量** | 受限于物理内存 |
| **生命周期** | 进程运行时保持 |

**工作原理**：
- 将已计算的KV Cache保留在内存中
- 下次遇到**相同前缀**的prompt时直接复用
- 避免重复计算

**在CS-Analyzer中的价值**：

```
场景：359通会话评分
├─ 每通会话的评分prompt包含：
│   ├─ 系统提示（固定内容，约1500 tokens）← 可缓存
│   ├─ 评分标准（固定内容，约1000 tokens）← 可缓存
│   └─ 会话内容（变化内容）← 不可缓存
│
├─ 热缓存50% = 缓存最近50%的KV Cache
│   → 系统提示和评分标准的KV Cache被保留
│   → 后续请求直接复用，跳过prefill阶段
│
└─ 效果：实测从24.6分钟 → 19.2分钟（提升22%）
```

### 2.3 冷缓存（Cold Cache / SSD缓存）

| 属性 | 说明 |
|------|------|
| **存储位置** | SSD硬盘 |
| **速度** | 较慢（但比重新计算快10-100倍） |
| **容量** | 很大（可达数百GB） |
| **生命周期** | 持久化保存，重启后仍在 |

**工作原理**：
- 内存放不下的KV Cache溢出到SSD
- 下次启动时从SSD恢复
- 适合**大模型、长上下文**

**在CS-Analyzer中的价值**：

```
场景：Qwen3.6-35B-A3B模型（20GB权重）
├─ 单请求KV Cache大小：
│   ├─ 短会话（1000 tokens）→ 约50MB
│   └─ 长会话（4000 tokens）→ 约200MB
│
├─ 359通会话总KV Cache：
│   └─ 约 50-100GB
│
├─ 热缓存50%（约64GB）→ 只能缓存部分
│   └─ 剩余部分如果丢弃 → 下次需重新计算
│
└─ 冷缓存20%（约371GB）→ 将溢出的KV Cache存到SSD
    → 下次启动时从SSD恢复
    → 避免重新计算prefill
```

## 四、定向优化方案

### 4.1 优化方向1：提升热缓存命中率

**当前问题**：
- 热缓存50% = 约64GB
- 但KV Cache可能更大，导致频繁溢出到SSD

**优化方案**：
```
1. 增大热缓存比例（如果内存允许）
   ├─ 当前：50%（64GB）
   └─ 建议：尝试70-80%（如果系统内存充足）

2. 优化prompt结构（提升前缀重复度）
   ├─ 当前：系统提示 + 评分标准 + 会话内容
   └─ 建议：将固定内容放入system prompt，user prompt只放变化部分
      → 前缀缓存命中率大幅提升
```

### 4.2 优化方向2：优化冷缓存使用

**当前问题**：
- 冷缓存20% = 约371GB
- 但SSD速度比内存慢10倍

**优化方案**：
```
1. 使用高速SSD（已满足，MacBook Pro内置SSD很快）

2. 定期清理冷缓存（避免过期数据占用空间）

3. 监控缓存命中率
   ├─ 在oMLX日志中搜索 "cache hit rate"
   └─ 如果命中率低，调整缓存大小
```

### 4.3 优化方向3：系统Prompt优化（高价值）

**基于Apple官方MLX文档的技术**：

```
当前prompt结构：
┌─ system prompt: 评分标准说明（固定，1500 tokens）
├─ user prompt:
│   ├─ 固定前缀：输出格式要求（固定，500 tokens）
│   └─ 变化内容：会话内容 + 规则（变化，2500 tokens）
└─ 效果：70%的prompt可被缓存，prefill时间减少70%
```

**预期收益**：
- prefill时间从30-60秒 → 10-20秒
- 总耗时从19.2分钟 → **14-16分钟**（预估）

## 五、讨论过程记录

### 5.1 金总的问题

> "oMLX的热缓存和冷缓存有什么作用？对于我们当前的系统又能带来什么价值？是否可以针对这两个做定向的优化？"

### 5.2 小虾米的调研

1. **搜索来源**：
   - Apple官方MLX文档（machinelearning.apple.com）
   - Apple Silicon LLM推理优化指南（blog.starmorph.com）
   - oMLX GitHub仓库
   - Reddit r/oMLX社区讨论

2. **关键发现**：
   - M5 Max内存带宽614 GB/s，是推理瓶颈
   - KV Cache是速度提升100倍的关键
   - oMLX的BlockAwarePrefixCache支持SSD前缀缓存
   - Apple官方确认M5神经加速器主要加速prefill阶段

### 5.3 讨论中的关键洞察

**关于并发关系的讨论**：

```
金总："既然厨房只有32个灶台，前台为什么还要派90单呢？"

小虾米："因为评分不是'炒菜'，是'炖汤'。"

解释：
- 评分请求在decode阶段GPU未满载
- 超额订阅（Over-subscription）确保GPU始终有活干
- KIMI=90是冗余保护，实际并发由Worker=30决定
```

**关于配置关系的讨论**：

```
oMLX=32（厨房32个灶台）
Worker=30（30个服务员）
KIMI=90（门口限流栏杆，防止挤爆）

实际并发 = min(30, 90, 32) = 30
```

## 七、总结

| 缓存类型 | 作用 | 当前配置 | 优化空间 |
|---------|------|---------|---------|
| **热缓存** | 内存中的KV Cache，最快 | 50% | 可尝试提升至70% |
| **冷缓存** | SSD中的KV Cache，持久化 | 20% | 当前足够 |
| **前缀缓存** | 复用相同prompt前缀的计算 | 已启用 | 优化prompt结构提升命中率 |

**核心结论**：
1. 缓存调整是性价比最高的优化（简单调整即可提升22%）
2. 系统Prompt优化是下一步高价值方向（预估再提升20-30%）
3. 高并发在Apple Silicon上有害（超过最优阈值后速度反而下降）
