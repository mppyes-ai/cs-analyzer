## 2. 根本原因拆解

### 🔴 根因1：Kimi API响应缓慢（主因）

**证据**:
```bash
# 最小范围测试（刚完成）
测试1-同步API: 38.6秒
测试2-异步API: 30.2秒
测试3-批量并发: 27.5秒（平均）
```

**结论**: 
- API本身响应时间30-40秒，比正常值（10-15秒）慢**2-3倍**
- 生产环境日志显示63-295秒，比测试还慢**2-7倍**
- **这是导致总耗时超长的主要原因**

**影响权重**: ~70%

---

### 🟡 根因2：Worker被多次中断重启（次因）

**证据**:
```bash
# Worker启动记录
grep "获取锁成功" /tmp/worker.log
✅ 获取锁成功 (PID: 74908)
✅ 获取锁成功 (PID: 76241)
✅ 获取锁成功 (PID: 87172)
✅ 获取锁成功 (PID: 87797)

# Worker退出记录
grep "收到退出信号" /tmp/worker.log
⚠️ 收到退出信号，正在保存当前任务...  [出现3次]

# Worker崩溃记录
grep "ModuleNotFoundError" /tmp/worker.log
ModuleNotFoundError: No module named 'dotenv'
```

**结论**:
- Worker启动了4次，正常应该1次
- 3次收到SIGINT/SIGTERM信号被中断
- 1次因`dotenv`模块缺失崩溃
- 每次重启需要重新加载模型（~30秒），累计浪费时间~2分钟

**影响权重**: ~20%

---

### 🟡 根因3：批量大小限制未生效（配置问题）

**证据**:
```python
# 代码显示限制20通/批
def calculate_adaptive_batch_size(...):
    return min(potential_size, 20, len(sessions))  # 硬编码限制20

# 但日志显示有50通/批的调用
grep "expected_count=50" /tmp/worker.log
[DEBUG] _call_kimi_async START - expected_count=50
```

**分析**:
- 代码限制20通/批，但日志显示有50通/批的API调用
- 这意味着`calculate_adaptive_batch_size`没有被正确调用，或配置被覆盖
- 大批量导致API响应时间更长（50通需要295秒 vs 20通需要217秒）

**影响权重**: ~10%

## 4. 结论

### ✅ 已确认的根本原因（按影响排序）

1. **Kimi API服务端响应缓慢**（主因，70%）
   - 单次API调用耗时30-40秒（测试）到63-295秒（生产）
   - 比正常值慢2-18倍
   - **这是不可控的外部因素**

2. **Worker进程被多次中断**（次因，20%）
   - 4次启动，3次被信号中断
   - 可能原因：父进程退出、系统信号、或手动干预
   - **需要进一步调查信号来源**

3. **批量大小限制未完全生效**（辅因，10%）
   - 代码限制20通/批，但日志显示50通/批调用
   - 可能配置被覆盖或代码路径不同

### ⚠️ 未发现的问题

- 不是代码逻辑错误导致循环
- 不是数据库死锁或阻塞
- 不是网络完全中断
- 不是内存不足导致交换
