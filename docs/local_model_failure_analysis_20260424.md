## 根因分析

### 1. 本地模型JSON输出问题

**云端模型 (kimi-k2.5)**: 43/43 成功 (100%)
**本地模型 (qwen3.6-35b-a3b)**: 38/43 成功 (88.4%)

本地模型在生成JSON评分结果时，部分结果缺少必要字段：
- `dimension_scores` 为 null 或缺失
- `summary` 为 null 或缺失

### 2. 具体失败模式

**批次3/5**（10通会话）：
- API调用成功，返回10个结果
- 但**10个全部失败**，说明模型虽然返回了JSON数组，但数组中的对象结构不完整

**重试批次1/3**（5通会话）：
- API调用成功，返回5个结果
- 只有**1个成功**，4个失败

**重试批次2/2**（4通会话）：
- API调用成功，返回4个结果
- **4个全部成功**，但出现 `conversion分数截断: 0 -> 1`

### 3. 模型行为差异

**本地模型特点**：
- 能正确理解评分任务
- 能返回JSON格式
- 但在**复杂会话**或**批量处理**时，部分结果的结构不完整
- 可能原因：
  1. 模型参数量较小(35B vs 云端大模型)，处理长上下文时注意力分散
  2. 量化精度损失(4bit)，影响输出质量
  3. 批量评分时，模型对每个会话的关注度不均

## 解决方案建议

### 方案1: 增强JSON验证和修复（推荐）

在 `_parse_batch_response` 方法中增加对不完整结果的自动修复：

```python
def _parse_batch_response(self, content: str, expected_count: int) -> List[Dict]:
    results = json.loads(cleaned)
    
    # 修复不完整的结果
    for i, result in enumerate(results):
        if not result.get('dimension_scores'):
            # 使用默认评分
            result['dimension_scores'] = {
                'professionalism': {'score': 3, 'reasoning': '评分缺失'},
                'standardization': {'score': 3, 'reasoning': '评分缺失'},
                'policy_execution': {'score': 3, 'reasoning': '评分缺失'},
                'conversion': {'score': 3, 'reasoning': '评分缺失'}
            }
        if not result.get('summary'):
            result['summary'] = {
                'total_score': 12,
                'risk_level': '中风险'
            }
    
    return results
```

### 方案2: 降级为单通处理

对于批量评分失败的任务，自动降级为单通评分（1通/批），提高成功率。

### 方案3: 使用云端模型兜底

配置双模型策略：
- 优先使用本地模型（节省成本）
- 本地模型失败时，自动切换到云端模型重试
