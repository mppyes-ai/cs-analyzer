# CS-Analyzer 客服会话质量分析系统 v2 完整介绍

## 一、系统定位与目标

**CS-Analyzer v2** 是一套**基于结构化规则知识库的智能客服质检平台**，专为璞康集团电商客服场景设计。

### 核心目标

| 目标 | 说明 |
|------|------|
| **质量评估** | 对客服会话进行4维度智能评分（4-20分制） |
| **问题识别** | 自动提取会话中的问题、亮点、改进建议 |
| **知识沉淀** | 通过人工矫正→规则提取→知识库更新，形成闭环 |
| **可解释性** | CoT（Chain-of-Thought）输出判定过程，质检员可理解"为什么扣这分" |

## 二、核心架构：混合双擎 + 三层知识

### 2.1 混合双擎存储架构

```
┌─────────────────────────────────────────────────────────────┐
│                     混合双擎架构                              │
├──────────────────────┬──────────────────────────────────────┤
│    SQLite 主数据层    │      LanceDB 向量索引层               │
├──────────────────────┼──────────────────────────────────────┤
│ • 结构化规则（22字段） │ • 复合文本向量（384维）               │
│ • 审核状态管理        │ • 语义相似度检索                      │
│ • 版本历史记录        │ • Metadata过滤（场景/维度/时效）      │
│ • 分析记录存储        │ • 指向SQLite的rule_id指针             │
└──────────────────────┴──────────────────────────────────────┘
                           ↓
                    查询流程：会话向量化 → LanceDB召回Top-K 
                    → SQLite拉取完整规则 → 喂给Kimi评分
```

**设计来源**：A&B反馈共识，解决"纯向量检索不稳定"和"纯结构化检索缺乏语义泛化"的问题。

### 2.2 三层知识系统

| 层级 | 内容 | 用途 | 回答什么问题 |
|------|------|------|-------------|
| **事实知识层** | 产品参数、安装要求、售后政策、活动政策 | 参数核对、政策查询 | "对不对" |
| **评分规则层** | 场景规则、评分标准、扣分条件、checkpoints | 质检评分依据 | "该怎么判" |
| **案例示例层** | 正例对话、反例对话、矫正案例 | Few-shot学习、培训材料 | "类似情况怎么判" |

## 四、技术实现详解

### 4.1 漏斗式意图分类（三层架构）

**问题背景**：日均3000+会话，全部走LLM成本太高且慢。

**解决方案**：

```
第一层：毫秒级规则匹配（拦截60-70%）
  ├─ 物流查询: "发什么快递"、"什么时候到"
  ├─ 发票相关: "发票"、"专票"、"普票"
  ├─ 尺寸规格: "多大"、"尺寸"
  ├─ 价格查询: "多少钱"、"价格"
  └─ 简单结束: "谢谢"、"好的"
  → <5ms，无需LLM

第二层：本地Qwen2.5:7b（复杂场景）
  └─ 客诉、多轮对话、情绪复杂、长尾场景
  → 1-3s，准确率高

第三层：关键词回退（保底）
  └─ 模型失败时使用规则兜底
  → 永不失败
```

**模型演进**：
- 最初尝试：Qwen3:4b → 强制思考模式，输出格式不可控，不适合API调用
- 最终采用：Qwen2.5:7b → 直接输出到response字段，适合API/结构化输出

### 4.2 混合检索（Hybrid Search）

**三层检索融合**：

```python
# 1. 全文检索（SQLite LIKE）- 精确匹配权重高
keyword_results = search_by_keywords(query)

# 2. 向量检索（LanceDB）- 语义泛化
vector_results = search_by_vector(query_embedding)

# 3. RRF融合排序（Reciprocal Rank Fusion）
final_results = rrf_fusion(keyword_results, vector_results, 
                          keyword_weight=0.4, vector_weight=0.6)
```

**权重配置**：向量权重0.6（语义理解），关键词权重0.4（精确匹配）。

### 4.3 CoT可解释评分

**Prompt设计要点**：

```json
{
  "dimension_scores": {
    "professionalism": {
      "score": 3,
      "reasoning": "判定过程：基于checkpoints逐项检查...",
      "evidence": ["证据片段1", "证据片段2"],
      "referenced_rules": ["rule_xxx"]
    }
  }
}
```

**前端展示**：
- 折叠面板展示判定过程
- 引用规则ID可点击查看详情
- 证据片段高亮显示

### 4.4 模型配置矩阵

| 模块 | 模型 | 位置 | 用途 | 选型理由 |
|------|------|------|------|----------|
| **意图分类** | Qwen2.5:7b + 规则 | 本地Ollama | 高频场景省成本 | 省60-70%算力 |
| **规则提取** | Kimi 2.5 | 远程API | 结构化规则生成 | 高质量JSON输出 |
| **智能评分** | Kimi 2.5 | 远程API | CoT推理 | 可解释性强 |
| **向量化** | paraphrase-multilingual | 本地 | 语义检索 | 384维，轻量 |

## 六、4维度评分体系

### 6.1 评分维度定义

| 维度 | 核心考察 | 5分标准 | 1分标准 |
|------|----------|---------|---------|
| **专业性** | 产品知识 | 参数准确、解释清晰 | 错误或无法回答 |
| **标准化** | 服务规范 | 礼貌用语、响应及时、结束规范 | 无礼貌、响应慢 |
| **政策执行** | 促销/售后 | 政策传达准确完整 | 政策错误或遗漏 |
| **转化能力** | 销售引导 | 主动挖掘需求、成功引导 | 无引导、用户流失 |

### 6.2 详细评分规则（以标准化为例）

**检查清单**：
- [ ] 首响时效：人工客服首条回复 ≤ 30秒
- [ ] 礼貌用语：包含"您好"、"请"、"谢谢"、"抱歉"
- [ ] 标准结束语：感谢+祝福+后续引导 三要素
- [ ] 响应连续性：不长时间中断（>3分钟需说明）
- [ ] 情绪管理：面对抱怨保持专业

**评分规则**：
- 5分：全部合格且优秀
- 4分：全部合格但有小瑕疵
- 3分：1-2项不合格
- 2分：3项不合格
- 1分：4项及以上不合格或态度恶劣

### 6.3 风险分级

| 总分 | 风险等级 | 颜色标识 | 处理建议 |
|------|----------|----------|----------|
| ≤8分 | 高风险 | 🔴 | 立即复盘，培训整改 |
| 9-12分 | 中风险 | 🟡 | 记录问题，定期改进 |
| ≥13分 | 正常 | 🟢 | 保持现有水平 |

## 八、工程实践与优化

### 8.1 异步任务队列 + 自动重试

```python
def run_worker():
    while running:
        # 1. 处理pending任务
        task = get_pending_task()
        if task:
            process_task(task)
            continue
        
        # 2. 无pending时，自动重试failed任务
        retried = retry_failed_tasks()
        if retried > 0:
            continue  # 立即处理重试的任务
        
        time.sleep(interval)
```

**实测效果**：22个会话，5个初始失败，自动重试后100%完成。

### 8.2 版本化管理（防止数据覆盖）

**问题**：原方案覆盖原分数，丢失历史，无法对比。

**解决方案**：
- `analysis_runs` 表独立存储分析历史
- `session_id` + `run_version` 联合唯一
- `is_active` 标记当前活跃版本
- 支持 `versions` 和 `compare` 命令查看历史

### 8.3 Golden Set评估体系

```bash
# 交互式标注
python golden_set_manager.py annotate --batch --limit 50

# 计算MAE（平均绝对误差）
python golden_set_manager.py mae

# 导出评估报告
python golden_set_manager.py export
```

**关键指标**：
- MAE：AI分vs人工分平均绝对误差（目标<1.0）
- Rule Recall@3：正确规则是否在前3条召回（目标>70%）
- Correction Reduction Rate：上线后人工改分率下降

### 8.4 工程规范（用户强制要求）

| 规范项 | 要求 | 理由 |
|--------|------|------|
| 数据库操作参数传递 | **必须使用关键字参数** | 字段位置变更时不影响调用 |
| Streamlit模块缓存 | 使用 `importlib.reload()` | 强制加载最新代码 |
| 字段变更流程 | 表结构+函数签名+调用点同步更新 | 避免运行时签名不匹配 |

## 十、系统亮点总结

1. **混合双擎架构**：兼顾结构化查询和语义检索，稳定性与泛化能力兼备
2. **漏斗式意图分类**：本地规则拦截60-70%高频场景，大幅降低算力成本
3. **CoT可解释评分**：质检员可理解AI评分依据，提升信任度
4. **闭环知识沉淀**：矫正→规则提取→审核→同步，知识库持续进化
5. **版本化管理**：分析历史可追溯，支持A/B对比和效果验证
6. **多层容错设计**：规则→LLM→关键词回退，单一故障不影响整体
