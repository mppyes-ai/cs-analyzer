## 问题列表

### 🔴 P0 - 阻塞性问题（影响核心功能）

#### #1 Worker重试机制未全自动运转
- **描述**: 第4轮出现超时后，Worker未及时自动重试，需人为提醒后才触发
- **证据**: 21:31:28首次超时，21:41:34才重试成功，间隔10分钟而非预期退避
- **根本原因**: 待排查
  - [ ] 检查retry_count是否正确递增
  - [ ] 检查重试间隔逻辑（固定vs指数退避）
  - [ ] 检查任务状态转换（processing→failed→pending）
- **状态**: 🟡 待排查
- **关联文件**: `worker.py`, `task_queue.py`

#### #2 进度计算错误+重复推送
- **描述**: 
  1. 60%后直接跳到100%（缺少80%）
  2. 同一进度多次重复推送（如20%出现两次）
- **证据**: 五轮测试均出现此现象
- **根本原因**: 待排查
  - [ ] Worker和MessagePoller是否同时推送
  - [ ] 进度计算公式是否正确（批次大小累加逻辑）
  - [ ] 60%跳跃是否与批次处理顺序有关
- **状态**: 🟡 待排查
- **关联文件**: `worker.py`, `message_poller.py`

### 🟠 P1 - 重要问题（影响性能/体验）

#### #3 超时重试的"成本"定义需澄清
- **描述**: 用户询问"成本高"具体指Token成本还是时间成本
- **问题**: 
  - 超时后重试是否会重复消耗Kimi Token（计费）？
  - 还是仅指时间成本（延迟）？
- **待验证**: 检查Kimi计费日志，同一请求ID重试时是否重复计费
- **状态**: 🔵 待回答
- **关联文档**: 第4轮Kimi计费日志

#### #4 动态超时机制设计
- **描述**: 超时时间应基于Token长度、预估耗时动态调整，而非固定300秒
- **背景**: 第4轮20通批次超时（300秒），重试后281秒成功，说明固定阈值不合理
- **待设计**: 
  - [ ] 公式：`timeout = max(300, estimated_tokens * 0.025)`
  - [ ] 或基于历史P95响应时间动态调整
- **状态**: 🔵 待设计
- **关联数据**: 五轮API指标数据

### 🟡 P2 - 一般问题（需要澄清/优化）

#### #5 Message Poller组件职责澄清
- **描述**: 用户此前未听说此组件，需解释其职责
- **说明**: MessagePoller是独立进程，负责：
  1. 监控分析任务进度
  2. 向飞书推送进度通知
  3. 与Worker并行运行
- **问题**: 可能与Worker的进度推送逻辑冲突，导致重复
- **状态**: 🔵 待解释
- **关联文件**: `message_poller.py`

#### #6 详细评分输出与时长关系
- **描述**: 如要求LLM输出更详细内容，会进一步放大耗时
- **背景**: Completion Tokens与API耗时正相关（r≈0.7）
- **待分析**: 
  - [ ] 当前评分输出的Token长度分布
  - [ ] 如增加详细度，预估会增加多少Token
- **状态**: 🔵 待分析
- **关联数据**: 五轮Completion Tokens数据
