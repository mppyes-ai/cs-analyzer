# 客服会话分析规则文档

## 1. 日志解析规则

### 1.1 文件格式
- 格式：`.log` 文本文件
- 编码：UTF-8

### 1.2 会话分隔

```
/******************以下为一通会话*********************/

jd_4f195a60996b3 2026-03-12 23:58:59
可以延迟发货吗
林内林小立 2026-03-12 23:59:09
人工客服【林小立】将为您竭诚服务。

/*******************会话结束_时间:2026-03-13 00:00:39*********************/
```

**格式说明：**

- 会话开始：`以下为一通会话`
- 会话结束：`会话结束_时间:YYYY-MM-DD HH:MM:SS`
- 消息格式：`用户ID 日期 时间`，消息内容在下一行

### 1.3 消息格式

```
用户ID YYYY-MM-DD HH:MM:SS
消息内容（可能跨多行）
```

**示例：**
```
jd_4f195a60996b3 2026-03-12 23:58:59
https://item.jd.com/100140537096.html
林内林小立 2026-03-12 23:59:09
人工客服【林小立】将为您竭诚服务。
```

### 1.4 角色识别规则

| 用户ID前缀 | 角色 | role值 |
|-----------|------|--------|
| `林内林` | 人工客服 | `staff` |
| `jimi_vender` | 机器人客服 | `staff` |
| 其他 | 用户 | `customer` |

### 1.5 特殊消息过滤
- 过滤内容：`【此消息为欢迎卡片或富文本模板答案，聊天记录中暂不支持展示】`
- 空消息：不存储

## 2. 多会话合并规则

### 2.1 合并条件
- 同一用户ID（第一个 customer 消息的 sender）
- 同一天内的多个独立会话

### 2.2 合并逻辑
1. 提取所有独立会话
2. 按用户ID分组
3. 每组按时间排序（从早到晚）
4. 合并消息列表

### 2.3 Session ID 生成
```
{用户ID}_{最早会话时间}

示例：
jd_4f195a60996b3_2026-03-12_235859
```

## 3. 消息存储格式

### 3.1 JSON 结构
```json
[
  {
    "sender": "jd_4f195a60996b3",
    "role": "customer",
    "timestamp": "2026-03-12 23:58:59",
    "content": "可以延迟发货吗"
  },
  {
    "sender": "林内林小立",
    "role": "staff",
    "timestamp": "2026-03-12 23:59:09",
    "content": "人工客服【林小立】将为您竭诚服务。"
  }
]
```

### 3.2 字段说明
| 字段 | 类型 | 说明 |
|------|------|------|
| sender | string | 发送者ID |
| role | string | `staff` 或 `customer` |
| timestamp | string | 发送时间 `YYYY-MM-DD HH:MM:SS` |
| content | string | 消息内容 |

## 4. 数据库存储规则

### 4.1 数据库设计原则

- **单表存储**：核心数据集中在 `sessions` 表
- **JSON存储**：消息列表、分析详情用JSON格式，灵活扩展
- **独立字段**：常用查询字段（摘要、评分）单独存储，便于索引和查询

### 4.2 表：sessions

```sql
CREATE TABLE sessions (
    session_id TEXT PRIMARY KEY,           -- 会话唯一ID
    user_id TEXT NOT NULL,                 -- 用户ID
    staff_name TEXT,                       -- 主要接待客服
    session_count INTEGER DEFAULT 1,       -- 合并的原始会话数
    start_time TEXT,                       -- 开始时间
    end_time TEXT,                         -- 结束时间
    created_at TIMESTAMP,                  -- 记录创建时间
    messages TEXT NOT NULL,                -- 消息列表（JSON）
    summary TEXT,                          -- 会话摘要
    professionalism_score INTEGER,         -- 专业性评分 1-5
    standardization_score INTEGER,         -- 标准化评分 1-5
    policy_execution_score INTEGER,        -- 政策执行评分 1-5
    conversion_score INTEGER,              -- 转化能力评分 1-5
    total_score INTEGER,                   -- 总分 4-20
    analysis_json TEXT                     -- 详细分析（JSON）
);
```

### 4.3 messages JSON 格式

```json
[
  {
    "sender": "jd_4f195a60996b3",
    "role": "customer",
    "timestamp": "2026-03-12 23:58:59",
    "content": "可以延迟发货吗"
  },
  {
    "sender": "林内林小立",
    "role": "staff",
    "timestamp": "2026-03-12 23:59:09",
    "content": "人工客服【林小立】将为您竭诚服务。"
  }
]
```

### 4.4 analysis_json JSON 格式

```json
{
  "issues": ["问题1", "问题2"],
  "suggestions": ["建议1", "建议2"],
  "dimensions": {
    "professionalism": {"score": 4, "reason": "..."},
    "standardization": {"score": 4, "reason": "..."},
    "policy": {"score": 3, "reason": "..."},
    "conversion": {"score": 3, "reason": "..."}
  }
}
```

### 4.5 表：daily_stats（每日统计）

```sql
CREATE TABLE daily_stats (
    date TEXT PRIMARY KEY,
    total_sessions INTEGER,
    avg_professionalism REAL,
    avg_standardization REAL,
    avg_policy_execution REAL,
    avg_conversion REAL,
    avg_total REAL,
    created_at TIMESTAMP
);
```

详细表结构、索引、查询示例见 `database_schema.md`



## 5. 质量分析规则

### 5.1 4维度评分体系

#### 专业性 (Professionalism) - 产品知识
| 分数 | 标准 |
|------|------|
| 5分 | 参数准确、解释清晰、专业术语使用正确 |
| 3分 | 基本正确但不完整 |
| 1分 | 错误或无法回答 |

#### 标准化 (Standardization) - 服务规范
| 分数 | 标准 |
|------|------|
| 5分 | 礼貌用语、响应及时、结束规范 |
| 3分 | 基本规范但有瑕疵 |
| 1分 | 无礼貌、响应慢 |

#### 政策执行 (Policy) - 促销/售后政策传达
| 分数 | 标准 |
|------|------|
| 5分 | 政策传达准确完整 |
| 3分 | 部分传达或有小错 |
| 1分 | 政策错误或遗漏 |

#### 转化能力 (Conversion) - 销售引导能力
| 分数 | 标准 |
|------|------|
| 5分 | 主动挖掘需求、成功引导 |
| 3分 | 被动回答、引导弱 |
| 1分 | 无引导、用户流失 |

### 5.2 详细评分规则（标准执行规范）

**重要提示**：以下规则为分析标准，执行时严格按照此规范评分，确保分析结果的一致性和可比性。

#### 5.2.1 专业性（Professionalism）详细规则

**评分维度**：产品知识准确性、技术解释能力、问题解决能力

**检查清单**：
- [ ] 产品参数准确无误（型号、规格、尺寸等）
- [ ] 技术原理解释清晰易懂
- [ ] 适用场景描述准确
- [ ] 竞品对比客观准确
- [ ] 不能回答的问题是否给出替代方案（如转接、记录后回复）

**具体判定标准**：

| 场景 | 5分表现 | 3分表现 | 1分表现 |
|------|---------|---------|---------|
| 产品参数咨询 | 准确回答参数，并补充相关建议 | 基本正确但缺少细节 | 参数错误或说"不清楚" |
| 安装问题 | 详细说明安装要求、尺寸、注意事项 | 说了基本要求但不够详细 | 无法回答或误导 |
| 故障排查 | 准确判断原因并给出解决方案 | 给出方案但不够精准 | 判断错误或无法解决 |
| 竞品对比 | 客观对比优缺点，突出本产品优势 | 简单对比，缺乏说服力 | 贬低竞品或无法对比 |

**扣分项（每项扣1分，最低1分）**：
- 产品参数错误
- 技术解释错误
- 适用场景说错
- 对不能回答的问题没有给出替代方案

---

#### 5.2.2 标准化（Standardization）详细规则

**评分维度**：礼貌用语、响应时效、结束规范、服务连续性

**检查清单**：
- [ ] **首响时效**：人工客服首条回复时间 ≤ 30秒
- [ ] **礼貌用语**：包含"您好"、"请"、"谢谢"、"抱歉"等礼貌词
- [ ] **标准结束语**：会话结束有规范的结束语（感谢+祝福+后续引导）
- [ ] **响应连续性**：不长时间中断（中断>3分钟需说明原因）
- [ ] **情绪管理**：面对用户抱怨/投诉时保持专业和耐心

**具体判定标准**：

| 检查项 | 合格标准 | 不合格表现 |
|--------|----------|------------|
| 首响时效 | ≤30秒得满分，30-60秒扣1分，>60秒扣2分 | 超过2分钟无响应 |
| 礼貌用语 | 至少2处礼貌用语 | 无问候语、无谢谢 |
| 结束语 | "感谢+祝福+后续引导"三要素齐全 | 只说"好的"或突然结束 |
| 响应连续性 | 无>3分钟中断 | 长时间不回复无说明 |
| 情绪管理 | 始终保持礼貌 | 与用户争执、态度不耐烦 |

**评分规则**：
- 5分：全部检查项合格，且表现优秀
- 4分：全部检查项合格，但有小瑕疵
- 3分：1-2项不合格
- 2分：3项不合格
- 1分：4项及以上不合格或态度恶劣

---

#### 5.2.3 政策执行（Policy）详细规则

**评分维度**：促销信息传达、售后政策说明、费用透明、物流政策

**检查清单**：
- [ ] **促销信息**：主动告知当前活动、优惠力度、有效期
- [ ] **售后政策**：明确说明保修期、退换货政策、质保范围
- [ ] **费用透明**：安装费、辅材费、增值服务费说明清晰
- [ ] **物流政策**：发货时间、配送方式、延迟发货方案
- [ ] **赠品政策**：赠品内容、领取条件、发放时间

**主动告知 vs 被动回答**：
- **5分标准**：用户未主动询问时，客服主动提及相关政策
- **3分标准**：用户询问后，完整准确回答
- **1分标准**：用户询问后仍说不清楚，或用户未问完全不提

**扣分项（每项扣1分，最低1分）**：
- 促销信息说错（活动内容、有效期）
- 保修期说错
- 费用说明不清（未提辅材费、安装费等）
- 赠品政策未说明

**特殊注意**：
- 用户在比价/犹豫时，必须主动告知促销活动（转化关键点）
- 用户提到安装时，必须说明费用构成

---

#### 5.2.4 转化能力（Conversion）详细规则

**评分维度**：需求挖掘、产品推荐、购买引导、成交促成

**检查清单**：
- [ ] **需求挖掘**：询问用途、房屋情况、预算范围、使用时间
- [ ] **产品推荐**：基于需求推荐合适型号，说明推荐理由
- [ ] **促销引导**：告知优惠、强调限时、催促下单
- [ ] **异议处理**：处理价格、功能、服务等方面的异议
- [ ] **跟进邀请**：留下联系方式、预约回访、添加微信等

**转化阶段判定**：

| 阶段 | 用户表现 | 客服应做 | 评分基准 |
|------|----------|----------|----------|
| 意向期 | 咨询产品信息 | 挖掘需求、推荐型号 | 是否主动推荐 |
| 比价期 | 询问价格、对比竞品 | 强调优势、告知优惠 | 是否突出性价比 |
| 决策期 | 询问细节、犹豫 | 解答疑虑、催促下单 | 是否推动决策 |
| 流失风险 | 说要考虑、要取消 | 挽留、了解原因、提供方案 | 是否有效挽留 |

**具体评分**：

| 场景 | 5分表现 | 3分表现 | 1分表现 |
|------|---------|---------|---------|
| 用户询价 | 询问用途→推荐型号→说明优惠→催促下单 | 回答价格，等待用户决定 | 只说价格，无后续 |
| 用户犹豫 | 了解顾虑→针对性解答→强调优势→促成 | 简单解释，不强求 | 不主动了解顾虑 |
| 用户要取消 | 询问原因→提供方案（如赠品、延期）→挽留 | 询问原因，简单挽留 | 直接同意取消 |

**流失判定**：
- 用户明确说"不要了"、"买别的了"、"取消订单" → 转化失败
- 客服未做任何挽留尝试 → 1分

---

### 5.3 特殊场景处理规则

#### 场景1：用户明确要取消订单
- **转化能力评分**：最高3分（无法转化，但可挽留）
- **评估重点**：
  - 是否询问取消原因？
  - 是否提供替代方案（如延期发货、换型号）？
  - 是否尝试挽留？
- **专业性和标准化**：正常评估

#### 场景2：投诉/争议
- **标准化评分**：重点关注态度和情绪管理
- **专业性评分**：重点关注问题解决能力
- **风险识别**：是否有升级风险？是否有后续跟进？
- **转化能力**：投诉处理不以转化为目标，不评估

#### 场景3：机器人转人工
- **分析范围**：只分析人工客服部分
- **评估重点**：
  - 人工是否解决机器人未解决的问题？
  - 是否了解之前的对话上下文？
  - 是否平滑衔接？

#### 场景4：闲聊/无效会话
- **判定标准**：用户无明确购买/咨询需求，纯闲聊
- **处理方式**：
  - 标记为"无效会话"
  - 不参与质量统计
  - 仅记录原始数据

#### 场景5：技术故障/系统问题
- **判定标准**：因系统问题导致服务中断
- **处理方式**：
  - 分析已发生的服务部分
  - 系统问题导致的延迟/中断不扣分
  - 备注说明

---

### 5.5 输出规范

**必须包含的字段**：
1. **4维度评分**：professionalism、standardization、policy、conversion
2. **评分理由**：每个维度的具体理由（必须引用会话中的具体表现）
3. **总分**：total_score（4-20分）
4. **摘要**：summary（一句话概括会话主题和结果）
5. **问题列表**：issues（具体可改进的问题）
6. **建议列表**：suggestions（具体可执行的改进建议）

**评分理由写作规范**：
- 必须引用具体表现，如"首响9秒，超过30秒标准"
- 必须说明依据的规则，如"根据政策执行规则第2条"
- 不可使用模糊描述，如"还行"、"不错"

**示例（好）**：
```
standardization: {
  "score": 4,
  "reason": "使用了礼貌用语（'您好'、'请'），结束语规范（'感谢您的咨询，祝您生活愉快'），但首响时间为9秒，虽在30秒标准内，但可更快"
}
```

**示例（不好）**：
```
standardization: {
  "score": 4,
  "reason": "服务态度还行，比较规范"
}
```

## 6. 工作流程

### 6.1 AI直接分析模式（推荐）

```
用户：小虾米，分析这个日志文件 xxx.log

AI：
1. 读取 .log 文件
2. 按规则解析会话
3. 按规则分析质量
4. 写入数据库
5. 返回分析报告
```

### 6.2 独立脚本模式（备用）

```
用户：python analyze_sessions.py xxx.log

脚本：
1. 解析 .log 文件
2. 提取原始消息
3. 写入数据库（评分字段默认为0）

后续：
用户：小虾米，帮我分析这些会话
AI：读取数据库 → 分析 → 更新评分和分析结果
```
